# Deployment Guide: Remote Job Launcher

This guide explains how to build and deploy the Remote Job Launcher application.
This application runs as a Deployment in one Kubernetes cluster (Cluster A) and is designed to launch Kubernetes Jobs in another Kubernetes cluster (Cluster B).

## Prerequisites

1.  **Two Kubernetes Clusters (or more):**
    *   **Cluster A (Source):** An EKS cluster where the Remote Job Launcher will be deployed.
    *   **Cluster B (Target):** An EKS cluster (or other Kubernetes cluster) where Jobs will be launched.
2.  **AWS CLI:** Installed and configured, with permissions to manage IAM roles and EKS clusters if setting up IRSA.
3.  **Docker:** Installed locally to build the Docker image for the launcher application.
4.  **kubectl:** Installed and configured for interacting with Cluster A and Cluster B.
5.  **Docker Registry:** A container registry (like Docker Hub, Amazon ECR, GCR, ACR) to store the built Docker image.
6.  **(Optional) Kubeconfig files:** For any target clusters that will be accessed using the `kubeconfig` authentication method.

## Build Steps

1.  **Navigate to the Application Directory:**
    ```bash
    cd /path/to/your/project/app
    ```

2.  **Build the Docker Image:**
    Replace `YOUR_DOCKER_REGISTRY/remote-job-launcher:latest` with your actual image repository and tag.
    ```bash
    docker build -t YOUR_DOCKER_REGISTRY/remote-job-launcher:latest .
    ```

3.  **Push the Docker Image:**
    ```bash
    docker push YOUR_DOCKER_REGISTRY/remote-job-launcher:latest
    ```

## Deployment Overview

The Remote Job Launcher supports two primary authentication methods for accessing target Kubernetes clusters:

1.  **IRSA (IAM Roles for Service Accounts):**
    *   **Recommended for EKS target clusters.**
    *   Leverages AWS IAM roles for secure, token-based authentication that automatically handles token refresh.
    *   Requires setting up specific IAM roles in AWS and configuring Cluster A's Service Account and Cluster B's `aws-auth` ConfigMap.
2.  **Kubeconfig File:**
    *   Suitable for non-EKS Kubernetes clusters or EKS clusters where IRSA is not feasible/desired (e.g., using long-lived IAM user credentials, which is generally not recommended for EKS).
    *   Involves mounting a kubeconfig file as a Kubernetes Secret in Cluster A.
    *   **Caution for EKS:** Standard EKS kubeconfigs generated by `aws eks update-kubeconfig` contain temporary tokens. Using these directly as static secrets will lead to authentication failures when the token expires. IRSA is the preferred method for EKS.

## I. Deployment using IRSA (for EKS Targets)

This is the recommended method for targeting EKS clusters.

**1. Configure IAM Roles and Policies (AWS):**
   *   You need to create two IAM roles:
        *   `ClusterALauncherRole`: In Cluster A's AWS account, assumed by the launcher's Service Account.
        *   `ClusterBJobExecutionRole`: In Cluster B's AWS account (if different, or can be same account), assumed by `ClusterALauncherRole`, and mapped in Cluster B's `aws-auth`.
   *   **Detailed instructions for creating these roles, their trust policies, and permission policies are in `docs/IAM_ROLES_GUIDE.md`. Please follow this guide carefully.**

**2. Update Service Account in Cluster A (`k8s/cluster-a/serviceaccount.yaml`):**
   *   Ensure the `remote-job-launcher-sa` ServiceAccount is annotated with the ARN of `ClusterALauncherRole`.
     ```yaml
     apiVersion: v1
     kind: ServiceAccount
     metadata:
       name: remote-job-launcher-sa
       namespace: default # Or your preferred namespace
       annotations:
         # Replace with the ARN of ClusterALauncherRole you created
         eks.amazonaws.com/role-arn: arn:aws:iam::CLUSTER_A_AWS_ACCOUNT_ID:role/ClusterALauncherRole
     ```
   *   Apply this change: `kubectl apply -f k8s/cluster-a/serviceaccount.yaml`

**3. Configure `aws-auth` ConfigMap in Target EKS Cluster (Cluster B):**
   *   The IAM role that will interact with Cluster B's Kubernetes API needs to be mapped to a Kubernetes user/group in Cluster B.
        *   If you are using `--target-eks-role-arn` (pointing to `ClusterBJobExecutionRole`), then `ClusterBJobExecutionRole`'s ARN must be mapped.
        *   If you are NOT using `--target-eks-role-arn` (meaning `ClusterALauncherRole` itself interacts with Cluster B), then `ClusterALauncherRole`'s ARN must be mapped.
   *   Edit the `aws-auth` ConfigMap in Cluster B:
     ```bash
     kubectl edit configmap aws-auth -n kube-system
     ```
   *   Add an entry to `mapRoles`. Replace placeholders with your actual values:
     ```yaml
     data:
       mapRoles: |
         # ... other existing role mappings ...
         - rolearn: arn:aws:iam::CLUSTER_B_AWS_ACCOUNT_ID:role/ClusterBJobExecutionRole # Or ClusterALauncherRole ARN
           username: remote-job-launcher # This Kubernetes username will be used in RBAC
           groups:
             - remote-job-launchers   # Example Kubernetes group
     ```
   *   Save the ConfigMap.

**4. Configure Kubernetes RBAC in Target EKS Cluster (Cluster B):**
   *   The Kubernetes user/group mapped above (e.g., `remote-job-launcher` / `remote-job-launchers`) needs permissions to create jobs in Cluster B.
   *   Create a file named `cluster-b-rbac.yaml` with the following content. Adjust `namespace` and `subjects` as needed.
     ```yaml
     # cluster-b-rbac.yaml - Apply to Cluster B
     apiVersion: rbac.authorization.k8s.io/v1
     kind: Role
     metadata:
       namespace: default # Namespace where jobs will be launched in Cluster B
       name: remote-job-creator-role
     rules:
     - apiGroups: ["batch"]
       resources: ["jobs"]
       verbs: ["create", "get", "list", "watch", "delete"]
     - apiGroups: [""]
       resources: ["pods"]
       verbs: ["get", "list", "watch"]
     ---
     apiVersion: rbac.authorization.k8s.io/v1
     kind: RoleBinding
     metadata:
       namespace: default # Must match Role's namespace
       name: remote-job-creator-binding
     subjects:
     # Bind to the group (recommended)
     - kind: Group
       name: remote-job-launchers # Matches group in aws-auth
       apiGroup: rbac.authorization.k8s.io
     # Or bind to the user
     # - kind: User
     #   name: remote-job-launcher # Matches username in aws-auth
     #   apiGroup: rbac.authorization.k8s.io
     roleRef:
       kind: Role
       name: remote-job-creator-role
       apiGroup: rbac.authorization.k8s.io
     ```
   *   Apply this to Cluster B: `kubectl apply -f cluster-b-rbac.yaml --context <cluster-b-context>`

**5. Update and Apply Deployment Manifest in Cluster A (`k8s/cluster-a/deployment.yaml`):**
   *   Open `k8s/cluster-a/deployment.yaml`.
   *   **Image:** Change `image: YOUR_DOCKER_REGISTRY/remote-job-launcher:latest` to your pushed image.
   *   **Args for IRSA:**
     ```yaml
     # ...
     args:
       - "--auth-method"
       - "irsa"
       - "--target-cluster-name"
       - "my-eks-target"                # Name of your EKS cluster in Cluster B
       - "--target-cluster-region"
       - "us-west-2"                    # AWS Region of Cluster B
       - "--job-image"
       - "busybox"
       - "--job-namespace"
       - "default"                      # Namespace in Cluster B for the job (must match RBAC)
       # Optional: If Cluster B is in a different AWS account or you need to assume a specific role
       # This ClusterBJobExecutionRole must trust ClusterALauncherRole for assumption.
       # - "--target-eks-role-arn"
       # - "arn:aws:iam::CLUSTER_B_AWS_ACCOUNT_ID:role/ClusterBJobExecutionRole"
     # ...
     ```
   *   **Volumes for Kubeconfig (Optional with IRSA):** If you are *only* using IRSA, you can remove the `kubeconfigs` volumeMount and volume definition from `deployment.yaml`. If you might use the kubeconfig method for other clusters, keep it, and ensure `optional: true` is set for the secret.
   *   Apply the RBAC for Cluster A (if not already done, or if changed for kubeconfig access):
     ```bash
     kubectl apply -f k8s/cluster-a/rbac.yaml
     ```
   *   Apply the deployment:
     ```bash
     kubectl apply -f k8s/cluster-a/deployment.yaml
     ```

## II. Deployment using Kubeconfig File

Use this method for non-EKS clusters or special EKS cases (see caution above).

**1. Prepare Kubeconfig File:**
   *   Obtain the kubeconfig file for the target cluster (Cluster B).

**2. Create Kubernetes Secret in Cluster A:**
   *   The application's `deployment.yaml` (as provided in `k8s/cluster-a/deployment.yaml`) expects a secret (e.g., named `other-cluster-kubeconfig`). The *key* within this secret's data must match the value you provide to `--target-cluster-name` when using `kubeconfig` auth.
   *   Example: If your `--target-cluster-name` is `my-non-eks-cluster`, create the secret:
     ```bash
     kubectl create secret generic other-cluster-kubeconfig \
       --from-file=my-non-eks-cluster=/path/to/your/non-eks-cluster.kubeconfig \
       -n default # Or your target namespace in Cluster A
     ```

**3. Update and Apply Deployment Manifest in Cluster A (`k8s/cluster-a/deployment.yaml`):**
   *   Open `k8s/cluster-a/deployment.yaml`.
   *   **Image:** Change `image: YOUR_DOCKER_REGISTRY/remote-job-launcher:latest` to your pushed image.
   *   **Service Account:** Ensure `serviceAccountName: remote-job-launcher-sa` is set. The IRSA annotations on the SA won't harm kubeconfig auth.
   *   **Args for Kubeconfig:**
     ```yaml
     # ...
     args:
       - "--auth-method"
       - "kubeconfig"
       - "--target-cluster-name"
       - "my-non-eks-cluster"  # Must match the key in the secret
       - "--job-image"
       - "busybox"
       - "--job-namespace"
       - "default"             # Namespace in Cluster B for the job
       # Optional: If your kubeconfig secret is mounted elsewhere than /etc/kubeconfigs/target-cluster-name
       # - "--kubeconfig-base-path"
       # - "/custom/kubeconfigs"
     # ...
     ```
   *   **Volumes:** Ensure the `kubeconfigs` volumeMount and volume are configured to mount the correct secret:
     ```yaml
     # ...
     volumeMounts:
     - name: kubeconfigs
       mountPath: "/etc/kubeconfigs" # Default base path
       readOnly: true
     volumes:
     - name: kubeconfigs
       secret:
         secretName: other-cluster-kubeconfig # Matches the secret you created
         # optional: false # If kubeconfig is the only method for this deployment, make it non-optional.
     # ...
     ```
   *   Apply Cluster A RBAC (grants launcher SA permission to read the kubeconfig secret):
     ```bash
     kubectl apply -f k8s/cluster-a/serviceaccount.yaml # Ensure SA exists
     kubectl apply -f k8s/cluster-a/rbac.yaml
     ```
     *Note: `k8s/cluster-a/rbac.yaml` currently refers to `resourceNames: ["cluster-b-kubeconfig"]`. Update this to match your secret name (e.g., `other-cluster-kubeconfig`) or make it more generic if you use multiple kubeconfig secrets.*
   *   Apply the deployment:
     ```bash
     kubectl apply -f k8s/cluster-a/deployment.yaml
     ```

**4. Configure Kubernetes RBAC in Target Cluster (Cluster B):**
    *   The identity defined within the kubeconfig file for Cluster B must have permissions to create jobs in the target namespace. This setup is external to this tool but crucial.

## III. Verification (Applicable to both methods)

1.  **Check Launcher Pods in Cluster A:**
    ```bash
    kubectl get pods -n <namespace-in-A> -l app=remote-job-launcher
    kubectl logs -f <launcher-pod-name> -n <namespace-in-A>
    ```
    Look for logs indicating successful connection and job creation attempts.

2.  **Check Job in Cluster B:**
    Switch your `kubectl` context to Cluster B.
    ```bash
    kubectl get jobs -n <job-namespace-in-B>
    kubectl get pods -n <job-namespace-in-B> # Look for pods created by the job
    kubectl logs -f <job-pod-name-in-B> -n <job-namespace-in-B>
    ```

## Troubleshooting

*   **IRSA Issues:**
    *   **"Unable to assume role" / "AccessDenied" in launcher logs:**
        *   Verify `ClusterALauncherRole` trust policy and OIDC provider.
        *   Verify `ClusterALauncherRole` has `sts:AssumeRole` permission for `ClusterBJobExecutionRole` (if used).
        *   Verify `ClusterBJobExecutionRole` trust policy allows `ClusterALauncherRole`.
        *   Ensure the `eks.amazonaws.com/role-arn` annotation on the SA in Cluster A is correct.
    *   **EKS API errors from Cluster B (e.g., "Unauthorized"):**
        *   Check `aws-auth` ConfigMap in Cluster B: Is the correct IAM role ARN mapped to a Kubernetes `username` and/or `groups`?
        *   Check Kubernetes RBAC in Cluster B: Does the mapped `username` or `group` have a `RoleBinding` or `ClusterRoleBinding` granting permissions to create jobs in the target namespace?
*   **Kubeconfig Issues:**
    *   **"Kubeconfig file not found":**
        *   Verify `secretName` in `deployment.yaml` (Cluster A) matches the created secret.
        *   Verify `--target-cluster-name` arg matches a key within the secret data.
        *   Check `k8s/cluster-a/rbac.yaml`: Does the `remote-job-launcher-sa` have permission to `get` the specific secret?
    *   **"Connection refused/timeout" to Cluster B:**
        *   Kubeconfig for Cluster B is incorrect (API server endpoint).
        *   Network connectivity issues between Cluster A pods and Cluster B API server.
    *   **Authorization errors from Cluster B:** The identity within the provided kubeconfig does not have job creation permissions in Cluster B.
*   **General:**
    *   **ImagePullBackOff (Cluster A):** Docker image name in `deployment.yaml` is incorrect or image is not accessible.
    *   **Job creation failed (API error in logs):** Logs from launcher pod should give Kubernetes API error details from Cluster B.

## Multiple Target Clusters / Auth Methods

*   **Multiple EKS Targets (IRSA):**
    *   `ClusterALauncherRole` needs `eks:DescribeCluster` for all target EKS clusters.
    *   If using cross-account (`--target-eks-role-arn`), `ClusterALauncherRole` needs `sts:AssumeRole` for each corresponding `ClusterBJobExecutionRole_N`.
    *   Each target EKS cluster (Cluster B, C, D...) needs its `aws-auth` and K8s RBAC configured for the IAM role that will authenticate to it.
    *   Run separate launcher deployments in Cluster A, each configured with args for a specific target.
*   **Mixed Targets (IRSA and Kubeconfig):**
    *   Configure IRSA as above for EKS targets.
    *   For kubeconfig targets, create the necessary secrets in Cluster A.
    *   The `k8s/cluster-a/rbac.yaml` should grant access to all kubeconfig secrets used.
    *   Run separate launcher deployments, configuring `--auth-method` and related args appropriately for each. The `kubeconfigs` volume in `deployment.yaml` can be made `optional: true` if not all deployments will use it.

This guide provides a comprehensive overview. Adjust paths, names, and configurations to fit your specific environment.
